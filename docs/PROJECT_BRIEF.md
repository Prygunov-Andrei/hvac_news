# Краткое описание проекта: HVAC News Platform

## 1. Общая концепция
Разработка специализированного новостного мультиязычного ресурса для HVAC-индустрии (Отопление, Вентиляция, Кондиционирование). 
Проект представляет собой веб-приложение с разделением на Backend (API) и Frontend (Клиентская часть).

**Языки локализации (MVP):** Русский, Английский, Немецкий, Португальский.

## 2. Технологический стек
*   **Backend:** Python, Django, Django REST Framework (DRF).
*   **Frontend:** Отдельное SPA/SSR приложение (React/Next.js).
*   **База данных:** PostgreSQL (используется по умолчанию).
*   **Хранилище медиа:** Локальное хранилище (с возможностью масштабирования до S3 в будущем).

---

## 3. Функциональные разделы

### 3.1. Раздел «Новости» (Лента)
Основной раздел сайта. Представляет собой бесконечную ленту публикаций (или пагинацию) с сортировкой по дате публикации (от новых к старым).

**Функционал:**
*   **Формат контента:** Текст, изображения, локальные видео (встроенные плееры).
*   **Механизм создания (ETL-процесс):**
    *   Администратор загружает **один архив или файл** (специально размеченный `.md` + папка с медиа).
    *   Система парсит файл, извлекает контент для каждого языка.
    *   Медиа-файлы (картинки, видео) сохраняются на сервере один раз и привязываются ко всем языковым версиям новости.
    *   В тексте автоматически расставляются ссылки на сохраненные медиа.
*   **Редактирование:** Полная замена записи (удаление старой -> загрузка новой версии файла).
*   **Планировщик:** Поле «Дата публикации». Если дата > текущей, новость скрыта до наступления времени.
*   **Отображение:** При переключении языка сайта контент новости меняется на соответствующий язык.

### 3.2. Раздел «Производители»
Статичная таблица/каталог компаний, сгруппированная по регионам.

**Структура данных:**
*   Наименование компании.
*   Бренды (связь "один-ко-многим", кликабельные ссылки на раздел Брендов).
*   Ссылки на сайт (до 3-х шт).
*   Краткое описание (мультиязычное).
*   Регион (для группировки).

**Управление:** 
Использование стандартной **Django Admin Panel**. Это позволит удобно выбирать регионы и привязывать бренды через выпадающие списки, не опасаясь ошибок в текстовых файлах.

### 3.3. Раздел «Бренды»
Каталог брендов, отсортированный по алфавиту. Является "другим разрезом" данных о производителях, но со своим уникальным описанием.

**Структура данных:**
*   Логотип.
*   Название бренда.
*   Производитель (ссылка на строку в таблице Производителей).
*   Краткое описание бренда (уникальное поле, мультиязычное).

**Управление:** 
Django Admin Panel.

### 3.4. Раздел «Новостные ресурсы»
Каталог полезных ссылок по теме.

**Структура данных:**
*   Логотип.
*   Название ресурса.
*   Ссылка (URL).
*   Краткое описание (мультиязычное).

## 4. Пользователи и Интерактив

### 4.1. Ролевая модель
1.  **Гость (Незарегистрированный):**
    *   Просмотр всех разделов.
    *   Чтение комментариев.
    *   Переключение языка.
    *   Отправка сообщения через форму обратной связи.
2.  **Пользователь (Зарегистрированный):**
    *   Все права Гостя.
    *   Добавление комментариев к новостям.
    *   Редактирование и удаление *своих* комментариев.
3.  **Администратор:**
    *   Загрузка/удаление новостей (через .md).
    *   Управление справочниками (Бренды, Производители).
    *   Модерация комментариев (удаление любых).

### 4.2. Регистрация и Авторизация
*   Вход/Регистрация по Email.

### 4.3. Обратная связь
*   Форма отправки сообщения администратору.
*   Защита: CAPTCHA.
*   Действие: Отправка email на адрес администратора.

## 5. Технические требования к формату новостей (.md)

*Необходима разработка спецификации формата файла.*

**Концепция "Жесткого каркаса":**
Файл должен содержать мета-информацию и блоки для каждого языка.
*Примерная логика:*
1.  Блок определений ассетов (ссылки на локальные файлы картинок/видео в архиве).
2.  Блок [RU] (Заголовок, Тело статьи).
3.  Блок [EN] (Title, Body).
и т.д.

Медиа-файлы не дублируются, а вставляются по ID или имени файла.
